---

## Purpose

Every significant product, architecture, or governance decision gets a numbered ADR. Future teams and AI agents understand **why**, not just **what**.

---

## Format

Each ADR includes:

- **Status:** Proposed | Accepted | Deprecated | Superseded
- **Context** — what problem existed
- **Decision** — what we chose
- **Reason** — why
- **Consequences** — tradeoffs, follow-ups

## When to Write an ADR

- New governance layer (like Nertura Core)
- AI pipeline architecture change
- Breaking API or schema decision
- Tier/billing model change
- Major UX paradigm shift
- Deprecation of a module

---

## When NOT to Write an ADR

- Bug fixes
- Copy tweaks
- Dependency bumps
- Routine migrations
